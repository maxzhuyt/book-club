# Plan 05 — Iterative critique loop (CD as persona referee)

## Core idea

The closed model generates a draft. A second pass with the open+CD model
produces a *critique* of that draft from the persona's vantage point —
specifically pointing out where the draft sounds unlike the persona. The
closed model then revises its own draft using the critique. Loop until a
fidelity check passes (or a max-iter cap is hit).

```
[Claude] → draft_0 ─┐
                    ├→ [Qwen+CD] critique_0 ─┐
                    │                         ├→ [Claude revises] → draft_1 ─┐
                    │                         │                                │
                    └─ accept if fidelity ≥ τ ┘     loop until τ met or k=K   │
                                                                              ▼
                                                                          final
```

This is a natural inversion of Plan 02 (Direction A): instead of the closed
model polishing the open model's draft, the open+CD model audits the closed
model's draft. The closed model keeps the reasoning lead; CD becomes a
persona judge with a written explanation.

## What stays untouched

- `simulation/src/cd/decoder.py`, `prompts.py`, `alpha.py`.

The critique is generated by calling `ContrastiveDecoder.generate()` with a
*critique* user message; the underlying CD machinery is unchanged.

## What gets added

## Per-turn pipeline

```python
draft = call_opus(system=persona_prompt, user=phase_user_msg)
for k in range(MAX_ITERS):           # MAX_ITERS = 2 typically
    critique = cd_decoder.generate(
        s_pos = persona_prompt,
        s_neg = generic_reviewer_prompt,
        user_msg = CRITIQUE_TEMPLATE.format(
            phase_user_msg=phase_user_msg,
            draft=draft,
        ),
        alpha = compute_alpha(persona.summary(), phase=phase),
    )["text"]
    if fidelity_ok(draft, persona, phase):
        break
    draft = call_opus(
        system=persona_prompt,
        user=REVISE_TEMPLATE.format(
            phase_user_msg=phase_user_msg,
            previous_draft=draft,
            critique=critique,
        ),
    )
```

### CRITIQUE_TEMPLATE

```
You are reviewing a draft that someone wrote claiming it was you. Your job
is to identify, specifically, ways the draft does NOT sound like you.

Original prompt you were given:
<<<
{phase_user_msg}
>>>

The draft (claimed to be your response):
<<<
{draft}
>>>

In 4-6 short bullet points, list the SPECIFIC places where this draft does
not match how you would actually write. Quote the offending phrases. For
each, say what you would have written instead — in your actual voice. Be
concrete. Do not summarize the draft. Do not be polite. Do not list things
that are fine.
```

The critique is *generated under CD*, so the bullet points themselves are
in the persona's voice and reference the persona's actual preferences.
This is the trick that makes the critique informative rather than generic.

### REVISE_TEMPLATE

```
You wrote this draft:
<<<
{previous_draft}
>>>

You then read it back and noted, in your own voice, what felt wrong:
<<<
{critique}
>>>

Now rewrite the draft. Apply every concrete change you noted. Do NOT add
content the critique did not ask for. Keep the response addressing the
original prompt.

Original prompt:
<<<
{phase_user_msg}
>>>
```

### fidelity_ok(draft, persona, phase)

A small classifier that decides whether to keep looping. Options, from
cheapest to most informative:

1. **Length and lexical match**: draft's avg sentence length and top-k
   word use match the persona's stat profile within tolerance.
2. **Embedding distance**: draft's SBERT embedding sits within radius r of
   the persona's CD reference bank from Plan 03's `reference_bank`.
3. **CD logprob delta**: same Score A as Plan 03 — once `sum(log p_pos -
   log p_neg)` crosses a per-persona threshold, accept.

Option 3 is the most principled and reuses Plan 03's scorer; if Plan 03
is built first, this is free. Otherwise option 2 is the recommended start.

## Cost and risk

- **Cost — per turn**: 1 closed-model draft + K × (1 CD critique + 1 closed
  revision). With K=1 (one critique + one revision), that is 2 closed calls
  and 1 CD call per turn. Roughly 2.5–3× a single-pass closed-model turn.
- **Cost — Phase 3 scaling**: 2 rounds × 8 personas × ~3 calls ≈ 48 calls
  per round of orchestrated revision. Manageable but not cheap.
- **Risk — critique sycophancy**: the CD model, even with a "be harsh"
  prompt, may produce mostly-positive critiques. Mitigation: enforce a
  minimum number of bullet points; reject critiques that contain none of
  the trigger words (*does not*, *would not*, *instead*).
- **Risk — instability over revisions**: each revision drifts the draft
  further from its original substance. Mitigation: cap K=1 by default;
  allow K=2 only when fidelity is well below threshold; preserve the
  Phase-1 commit so later phases can't rewrite stance.
- **Risk — CD critique is itself persona-drifted on long inputs**: the
  open model's context is loaded with `phase_user_msg` and `draft`, both
  potentially long. CD's persona signal is strongest on short prefixes.
  Mitigation: keep critique max_new_tokens small (~200) and use α
  cautiously here — α=1.0 is fine; do not push to the cap.

## When this is worth using

- When per-turn cost is acceptable (this is the most expensive plan).
- When a downstream consumer needs strong persona fidelity AND coherent
  reasoning, and neither Plan 01 nor Plan 03 alone clears the bar.
- As a quality ceiling baseline for evaluation: "what's the best we can
  squeeze from a closed model with an open+CD referee in the loop?"

## How to evaluate

Same metrics as Plan 01. Additional plots:
- Fidelity score vs. iteration k (does it monotonically improve, or does it
  start drifting after k=1?).
- Fraction of turns that needed any revision (if < 20%, the loop isn't
  earning its cost).
- Substantive content overlap between draft_0 and draft_K (high = revision
  is purely stylistic; low = revision is changing the argument, which is
  a red flag).
